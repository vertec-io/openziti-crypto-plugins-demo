# OpenZiti Crypto Plugins Demo — prebuilt-images branch

**Fast path — pull pre-built images from GHCR and run the 9-cell interop matrix
in ~15 minutes.**

This branch is a companion to [`main`](../../tree/main). It hosts the same
nine-cell cipher-negotiation demonstration but replaces the five `docker
compose build` steps with `image:` references to pre-built container
images published at the GitHub Container Registry location recorded in
[image-manifest.txt](image-manifest.txt).

If you land here first: see [What differs from main](#what-differs-from-main)
and [Which branch should I use](#which-branch-should-i-use) below.

## Quick Start

```bash
git clone -b prebuilt-images https://github.com/vertec-io/openziti-crypto-plugins-demo
cd openziti-crypto-plugins-demo
docker compose pull
docker compose up -d
./scripts/wait-for-ready.sh
./runmatrix.sh --all
```

That is the full invocation — no `--build` flag, no profile builds. The
five images (controller, router, go / c / jvm samples) arrive by
`docker compose pull` and are already wired into the compose file by
the SHA-pinned tags shown in [image-manifest.txt](image-manifest.txt).

Results land in `evidence/` as per-cell outputs plus a `matrix-summary.txt`
with verdicts and timings. Teardown:

```bash
docker compose down -v
```

## What differs from main

This branch tracks main's sample code, cell params, scripts, and reference
plugin — the ONLY differences are `docker-compose.yml` (uses `image:` refs)
and `image-manifest.txt` (records which main SHA produced the pulled
images).

No Dockerfile changes, no source-tree changes, no matrix behaviour
difference. The two branches land identical per-cell verdicts when run on
the same published images — only wall-clock timing differs.

## Which branch should I use

- **[`main`](../../tree/main)** — audit the full build chain from source.
  First run 40-70 minutes; subsequent matrix runs under 5 minutes.
- **`prebuilt-images`** (you are here) — run the demo quickly and trust
  the published images' chain of custody. First run under 15 minutes; every
  image is a content-addressed pull from GHCR.

Choose main if you want to audit the full build chain from source. Choose
prebuilt-images if you want to run the demo quickly and trust the published
images' chain of custody.

## Image Provenance

Every image in [docker-compose.yml](docker-compose.yml) is pinned to a
content-addressed digest recorded in [image-manifest.txt](image-manifest.txt).
The manifest lists five lines in the format
`<name>=sha-<main-SHA>=sha256:<digest>`, one per image. These digests match
the output of `docker buildx imagetools inspect
ghcr.io/vertec-io/openziti-crypto-plugins-demo/<name>@sha256:<digest>`.

Images were built and scanned by `.github/workflows/publish-images.yml` on
the [main](../../tree/main) branch of this repository, which runs on every
push to main and emits the manifest as a workflow artifact.

To independently verify a manifest entry:

```bash
docker pull ghcr.io/vertec-io/openziti-crypto-plugins-demo/ziti-controller:sha-<main-SHA>
docker inspect --format '{{index .RepoDigests 0}}' \
  ghcr.io/vertec-io/openziti-crypto-plugins-demo/ziti-controller:sha-<main-SHA>
# Compare the sha256:... suffix with the ziti-controller line in image-manifest.txt
```

Repeat for the other four images. Any mismatch indicates the tag was
repointed after the manifest was captured; in that case the manifest must
be regenerated by re-running the publish workflow.

## Peer Variants

Every SDK (Go, C, JVM) is built in two configurations. Together with the
reference plugin, this gives three runtime variants:

| Variant    | Source                                          | Cipher Behaviour                       |
|------------|-------------------------------------------------|----------------------------------------|
| **stock**    | Upstream `openziti/*` latest stable           | Default negotiation only               |
| **hook-off** | Hook-enabled fork, default config             | Hook extension compiled in but unused  |
| **hook-on**  | Hook-enabled fork + reference plugin          | Alternate cipher registered and offered |

*hook-off* proves the fork is a no-op when no plugin is loaded.
*hook-on* proves a plugin can register an alternate cipher and negotiate it.

## Interop Matrix

The matrix exercises nine cells across five groups. Each group isolates one
property of the hook architecture.

| Group             | Cell                   | Client        | Host          | Expected Outcome       |
|-------------------|------------------------|---------------|---------------|------------------------|
| **Baseline**      | `baseline-go`          | stock-go      | stock-go      | Default cipher         |
|                   | `baseline-c`           | stock-c       | stock-c       | Default cipher         |
|                   | `baseline-jvm`         | stock-jvm     | stock-jvm     | Default cipher         |
| **Hook neutrality** | `neutrality-go`      | hook-off-go   | hook-off-go   | Default cipher         |
|                   | `neutrality-c`         | hook-off-c    | hook-off-c    | Default cipher         |
|                   | `neutrality-jvm`       | hook-off-jvm  | hook-off-jvm  | Default cipher         |
| **Matched**       | `matched-go`           | hook-on-go    | hook-on-go    | Alternate cipher (AES-256-GCM) |
| **Mismatched**    | `mismatched-go`        | hook-on-go (id 2) | hook-on-go (id 99) | Both sides reject |
| **Fallback**      | `fallback-go`          | hook-on-go    | stock-go      | Graceful fallback to default |

**What each group proves:**

- **Baseline** — unmodified upstream binaries interoperate normally.
- **Hook neutrality** — the fork changes are invisible when no plugin is
  loaded; behaviour is identical to stock.
- **Matched** — when both peers offer the same alternate cipher,
  negotiation selects it and data flows end-to-end.
- **Mismatched** — when peers offer incompatible ciphers, both sides
  reject cleanly with no data corruption.
- **Fallback** — a hook-enabled peer degrades gracefully when its
  counterpart does not support negotiation.

## What This Fork Changes

The hook-enabled forks add extension hooks to three areas of the OpenZiti
data path. Each change maps to a documented proposal PR:

**Controller + router (`vertec-io/ziti`, 4 commits, ~280 lines changed):**

- *PR 3* — Forward a cipher-preferences header through edge-router dial
  paths so that the hosting SDK can see what the client offered.
- *PR 7* — Add a controller startup lifecycle notifier so that plugins
  can register themselves before the first enrollment completes.
- *PR 2 consumer path* — Thread the cipher-provider interface through
  the router xgress setup so the negotiated provider is available at
  encryption time.
- *Build pin* — Lock `go.mod` dependencies to tagged fork versions of
  `secretstream` and `sdk-golang` so the build is reproducible without
  sibling checkouts.

**Go SDK (`vertec-io/sdk-golang`, 7 commits) and core library
(`vertec-io/secretstream`, 3 commits):**

- Introduce a `CryptoProvider` seam in `secretstream` for pluggable AEAD
  backends, with a default provider that preserves legacy behaviour.
- Surface cipher-negotiation failures as named errors.
- Emit the negotiated cipher id in structured logs.
- Add a `PreStartPolicy` gate so external code can intercept context
  construction.

**C SDK (`vertec-io/ziti-sdk-c`, 10 commits, ~900 lines changed):**

- Add a pluggable AEAD vtable and a CMake backend slot for alternate
  crypto implementations.
- Mirror the Go negotiation flow: send cipher preferences on dial,
  echo on reply, validate on both sides.
- Rename internal crypto-backend enum values to neutral identifiers.

All fork branches are named `feature/crypto-extensibility` and are pinned
to specific SHAs recorded in the main branch's Dockerfiles.

## Trace Contract

Every sample binary accepts a `--print-cipher` flag. On a successful
handshake it emits exactly one line to stdout:

```
NEGOTIATED-CIPHER:<id>
```

where `<id>` is a decimal integer identifying the negotiated cipher. The
harness scripts parse this line to determine the verdict for each cell:

- **PASS** — the observed cipher id matches the expected outcome.
- **REJECT-AS-EXPECTED** — both sides refused the handshake, which is the
  correct behaviour for the mismatched cell.
- **FAIL** — unexpected cipher id or missing trace line.

## Staying in sync with main

Every successful publish of images from [main](../../tree/main) is
automatically followed by a pull request against this branch that updates
`docker-compose.yml`'s tags and the digests in `image-manifest.txt`. The PR
title is `Sync prebuilt-images to main@<sha>`. Merging the PR brings this
branch forward to the newest published images.

If two main pushes land in quick succession, two sync PRs may open. The
convention is to squash-merge the newer PR and close the older one.
